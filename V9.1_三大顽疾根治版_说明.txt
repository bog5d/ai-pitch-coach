【这版解决了什么？】

V9.0 由 Cursor（AI 搭架子工具）快速搭建，功能齐全但留下了三个深层 Bug，
一个比一个顽固。V9.1 是总工程师级的外科手术——逐一根治，一条都不放过。


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  顽疾一（已根治）：点「开始生成复盘报告」直接崩溃
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【症状】
  您在狙击清单里写好了找茬疑点，兴冲冲点「开始生成」，
  系统直接报红：
    The truth value of a DataFrame is ambiguous. Use a.empty, a.bool()...

【是谁的锅】
  狙击清单返回值是一个「表格对象（DataFrame）」。
  旧代码用 Python 的「或」运算（or）来判断「有没有数据」——
  而对表格对象用「或」就像对一个装满了东西的盒子问「这个盒子存在吗」，
  Python 会直接懵掉、抛出错误。

【怎么修的】
  把「或」判断改为「是不是空的（is None）」判断：
  只要表格对象存在（哪怕是空表格），就直接使用，不再绕弯问真假。

【注意点】
  ✅ 现在无论您填没填狙击清单，点按钮都不会崩溃
  ✅ 多行狙击目标完整保留，一条不丢
  ✅ 不影响任何其他功能


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  顽疾二（已根治）：已有转写缓存，还是先压缩音频再查缓存
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【症状】
  明明「历史已转写」的绿色提示出来了，但控制台先显示的是
    「FFmpeg 极致压缩完成！新大小：7.24MB（体积缩减 93.8%）」
  先白白把一个 116MB 的大文件压缩一遍，再发现「哦原来有缓存」。
  这一压就是几十秒 CPU 全力跑，电脑嗡嗡响，纯纯的冤枉算力。

【是谁的锅】
  旧流水线的执行顺序是：
    读文件 → 【压缩音频】 → 算指纹(MD5) → 查缓存 → 发现有缓存，结束
  指纹可以从原始文件直接算，完全不需要等压缩完再算。
  只是因为代码写的顺序不对，导致「查缓存」这一步被排到了「压缩」之后。

【怎么修的】
  重排执行顺序：
    读文件 → 【算指纹】 → 【查内存缓存】 → 【查磁盘缓存】
      ↓ 缓存命中：直接用，全程跳过 FFmpeg
      ↓ 缓存未命中：才启动 FFmpeg 压缩 → 云端 ASR → 入库

  同时加入「阅后即焚」机制：
    FFmpeg 压缩完、ASR 转写入库之后，立刻自动删除临时压缩文件
    （之前每跑一次就留一个 _v62_asr_gateway.mp3，批量分析时会慢慢撑爆硬盘）

【注意点】
  ✅ 有缓存 → 秒级完成，完全不碰 FFmpeg，CPU 安静如初
  ✅ 无缓存大文件（≥10MB）→ 正常走 FFmpeg 压缩后再转写，体验不变
  ✅ 无缓存小文件（<10MB）→ 直通 ASR，从不压缩，行为与之前一致
  ✅ 临时压缩音频 ASR 完即删，硬盘不再悄悄积累垃圾文件
  ⚠️ 缓存 key 用的是「原始文件的 MD5 指纹」，所以同一个音频文件
     无论第几次点按钮，只要文件没变，就永远命中缓存


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  顽疾三（已根治）：提取文字稿像面条，没标点没换行
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【症状】
  点「提取文字稿」，出来的是：
    [发言人 1]: 先聊着啊就那个5万年8月是的那个来讲的就是最早他们跟他们一起在韩国
    威那边在过了就一起好像是个上次去一起对我们是曾请亚威然后去一起去了汉堡威
    最后在创立了德天昨天之航……（几千字不间断）
  一坨流水账，没逗号没句号，不同的人说的话挤在一起，根本没法看。

【是谁的锅（三层叠加）】

  第一层：标点从哪来？
    无论是硅基流动还是阿里云，「词级（word-level）」数据里是没有标点的。
    标点在「句子级（sentence/segment）」文字里——
    但旧代码只从词级抓数据，所以标点全部丢失。

  第二层：谁在说话从哪来？
    硅基流动（主引擎）根本不支持「说话人分离（Speaker Diarization）」，
    它不知道谁是投资人、谁是被访谈人，所有词都打成「说话人1」，
    结果就是一大段没有分割的语流。
    阿里云（备用引擎）是支持说话人分离的，但旧代码忘了在 API 请求里
    加一行开关参数 diarization_enabled=True，导致功能一直没启用。

  第三层：格式化逻辑不拆句
    即使词里有标点，旧格式化函数也只在「换人说话」时才换行，
    同一个人说了一万字也是一坨，没有自然段落感。

【怎么修的（三层对症下药）】

  针对第一层（标点补回来）：
    解析时主动读句子级文字，把句末标点（。？！，…）
    追加到这句话最后一个词的文字字段里。
    这样词里就带着标点了，后续展示时自然有停顿感。

  针对第二层（说话人分离）：
    阿里云请求参数加上 diarization_enabled=True，
    让服务端返回每个词属于哪位说话人，不同人的话自然分段。
    硅基流动暂不支持此功能，在该引擎下仍为单说话人展示，
    但至少标点已经有了。

  针对第三层（段内自动换行）：
    格式化函数升级：同一说话人内部，每当遇到句末标点（。？！…）
    就在当前句末插入换行，不再把整段话挤成一行。

【注意点】
  ✅ 阿里云兜底引擎：标点 + 说话人分离均已修复，体验最佳
  ✅ 硅基流动主引擎：标点已修复，说话人分离需等该平台未来支持
  ✅ 对齐机制完全不受影响：词级时间戳（start_time/end_time/word_index）
     100% 原样保留，点击音频跳转功能依然精确
  ✅ 报告生成流程（LLM 评委、风险点定位）不受此修改影响
  ⚠️ 文字稿展示质量取决于 ASR 引擎能力：
     硅基流动免费版对专业术语识别率有限，遇到军工、医疗、金融
     等高密度专业词汇时，建议用「热词列表」功能辅助提升识别率

【高管错题本 · 数据飞轮】（V8.6 引入，V9.1 保留）
  在审查台锁定导出后，系统自动对比初稿与终稿的差异。
  改动达到一定幅度（编辑距离>10%或字数差>10），
  就自动提炼一条记忆存入该公司该高管的「错题本」。
  下次同公司同被访谈人，评委 AI 会提前读取最多5条历史教训。

  配置要求：需要填写 DEEPSEEK_API_KEY（静默提炼与主评委走同一底座）

【三级 ASR 缓存】（V8.0 引入，V9.1 保留）
  内存缓存 → 磁盘缓存 → 云端转写，三层递进。
  同一音频文件只要转写过一次，后续复盘无需重新调用云端 API。
  V9.1 新增：缓存命中时彻底跳过 FFmpeg，体验更流畅。

【狙击清单（找茬模式）】（V7.5 引入，V9.1 保留）
  可在每条录音旁填写「原文引用」+「找茬疑点」，
  让评委 AI 对您预先标记的可疑话术进行重点审查。
  V9.1 新增：填写后点按钮不再崩溃（修复顽疾一）。

【审查台 + 人工精炼】（V7.5/V8.0 引入，V9.1 保留）
  AI 初稿 → 人工审查台 → 逐条确认/修改/删除 → 锁定导出 HTML 报告。
  支持单条风险点「局部精炼」（再召唤 AI 二次加工）。
