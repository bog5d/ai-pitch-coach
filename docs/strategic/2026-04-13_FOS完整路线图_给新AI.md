## 一、主理人的核心愿景（勿忘）

> **「接触一家新公司的第1秒，FOS就开始工作；融资关单的最后1分钟，FOS依然在工作。」**

具体来说：
1. 新公司丢来一个大杂烩文件夹 → 系统自动扫描、分类、打标签、识别缺口、生成「融资准备度评分」
2. 平时材料管理 → FSS资产台账持续维护（已建）
3. 见机构前 → 会前简报（已建）
4. 见机构后 → 访谈录音上传、AI分析、错题本积累（已建）
5. 找机构 → 基于已有机构画像 + 公开数据，自动发现最匹配的投资人
6. 追踪进度 → 每家机构的接触状态、下一步行动、Pipeline全览

这不是单个工具，是**融资全生命周期AI作战系统**。

## 三、待建模块完整路线图

### Sprint 1：材料体检引擎（阶段0-A）
**优先级**：🔴 最高  
**目标**：新公司扔来一个大杂烩文件夹，系统扫描后告诉你「还缺什么」  
**工期**：3-4天  
**位置**：FSS系统（`AI_CangJie_FSS/`）  

**需要新建的文件**：
```
AI_CangJie_FSS/
├─ src/
│    ├─ standard_checklist.json   ← 标准尽调清单（37类材料定义）
│    ├─ document_intake.py        ← 核心引擎：扫描、分类、缺口识别
│    └─ intake_llm_classifier.py  ← LLM文件分类（可选，先做关键词匹配）
└─ tests/
     └─ test_document_intake.py   ← 测试（全Mock，不读真实文件）
```

**核心逻辑**：
```python
# 1. 扫描文件夹
files = scan_directory(input_dir)  # → [{"filename": ..., "extension": ...}]

# 2. 分类每个文件
for f in files:
    category = classify_file(f)  # 先用关键词规则，后期可接LLM
    # 关键词规则示例：
    # "审计" → 财务审计; "营业执照" → 工商资质; "章程" → 公司章程

# 3. 缺口识别
checklist = load_standard_checklist()
present = {f['category'] for f in classified_files}
gaps = [item for item in checklist if item['category'] not in present]

# 4. 生成报告
report = generate_health_report(classified_files, gaps)
# 输出：准备度分数 + 已有文件清单 + 缺口清单
```

**输出示例**：
```
===== 融资材料准备度报告 =====
综合评分：62/100

✅ 已具备（8类）：
  - 商业计划书 ✓ (3个文件)
  - 财务审计报告 ✓ (1个文件，2024年度)
  - 营业执照 ✓
  ...

❌ 缺失（5类）：
  - 近3年完整三表（仅有2024年，缺2022/2023）
  - 股权结构图（图形版）
  - 核心团队简历（人力资源背景调查用）
  - 专利证书（知识产权证明）
  - 完整客户合同样本

⚠️ 建议优先补充（融资中最常被问的3项）：
  1. 近3年完整财务三表（PE/VC必问）
  2. 股权结构穿透图（法务尽调第一项）
  3. 核心团队简历（高管背景调查）
```

**测试要求**：
- `test_classify_file_by_keyword`：纯关键词规则，无IO
- `test_gap_analysis`：给定已有文件列表，验证缺口识别正确
- `test_health_score_calculation`：验证分数计算逻辑
- `test_scan_empty_dir`：边界case，空文件夹不崩溃
- 全部用mock，不读写真实文件系统

## 四、给下一任AI的技术规范

### 4.1 FSS系统规范

**绝对不能动的地方**：
- `资产控制台主程序.py` 里的 `向上扫描()` 和 `向下同步()` 核心逻辑
- Excel读写逻辑（基于openpyxl，严禁pandas）
- 现有GUI布局（tkinter，修改需特别小心）

**新功能添加规范**：
- 在 `src/` 子目录新建模块（现在可能还没有src目录，需要创建）
- 测试放 `tests/` 目录
- 只用标准库 + openpyxl（不引入新依赖）
- LLM调用：只在体检分类时用，且必须可降级到纯关键词规则

### 4.2 AI Coach系统规范

参见 `CLAUDE.md`（四大铁律）和 `AGENTS.md`（完整握手区）。  
关键：**先写测试，再写代码；改schema必须六向联动**。

### 4.3 数据流规范

```
FSS向上扫描 → asset_index.json → AI Coach asset_bridge.py → 会前简报
FSS体检引擎 → health_report.json → （未来）AI Coach材料准备度提示
AI Coach机构画像 → .company_profiles/*.json → investor_matcher.py
AI Coach访谈记录 → .executive_memory/ → pipeline_tracker.py（关联）
```

## 六、重要决策备忘

| 决策 | 结论 | 原因 |
|------|------|------|
| 体检引擎放哪里 | FSS系统 | 本质是文件操作，与FSS现有职责一致 |
| 匹配引擎放哪里 | AI Coach系统 | 依赖机构画像数据，已在AI Coach积累 |
| 先用规则还是LLM分类 | 先规则（关键词匹配），后LLM | 规则零成本、快速可测试；LLM作为升级选项 |
| 是否引入向量数据库 | 否 | 几百个文件用关键词匹配足够 |
| FSS是否迁移Web | 否（长期） | 保持EXE稳定性，统一入口是未来的事 |
| 开发顺序 | Sprint 1→2→3→4→5→6 | 每步完成才开始下一步，避免相互污染 |
