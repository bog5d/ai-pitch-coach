## 一、背景：这两个系统是怎么来的

主理人所在公司（泽天智航）正处于主动融资阶段，历经多轮与投资机构的交互后，亲身感受到以下痛点，并开始用AI协作开发工具来解决：

**痛点一**：高管访谈前没有针对性准备，不知道机构关注点在哪，不知道哪些问题会成为"翻车点"  
**痛点二**：每次给投资机构打包尽调材料要花1-2小时，人工翻找、复制、整理，效率极低  
**痛点三**：公司内部知识资产（BP、财务模型、尽调材料）没有系统化沉淀，每年融资几乎从零开始  
**痛点四**：团队成员对知识资产的贡献没有量化指标，无法与考核挂钩  

由此，主理人先后开发了两套工具，均采用**「主理人提需求，AI实现」**的协作模式：


| 系统  | 名称                     | 当前状态                        |
| --- | ---------------------- | --------------------------- |
| 系统A | AI路演教练（AI Pitch Coach） | 活跃开发，V10.3，513个测试通过         |
| 系统B | 仓颉FSS（资产台账管理系统）        | 双向同步引擎成熟，MatchMaker V4.0已搁置 |

## 二、融资工作流的五个阶段（主理人亲历版）

这是理解系统价值的基础，也是整合架构设计的坐标系。

```
阶段一：路演（线上/线下）
  - 动作：分享商业计划书(BP)，初步回答机构疑问和关注点
  - 痛点：不知道具体机构的投资偏好，准备不够针对性
  - 现有覆盖：部分（AI Coach的机构画像/会前简报功能）

阶段二：NDA签署 + 尽调材料清单
  - 动作：机构发来材料清单（微信/Excel/PDF），公司打包对应材料发送
  - 痛点：材料散落各处，打包耗时；材料质量参差，存在过时版本
  - 现有覆盖：尝试中（仓颉FSS MatchMaker V4.0，已搁置）

阶段三：内部高管访谈（立项后）
  - 动作：投资机构负责人深度访谈公司高管（CEO/CFO/CTO等）
  - 痛点：高管无法预判被问什么，真实回答中的"翻车点"无法事前发现
  - 现有覆盖：✅ 主战场（AI Coach核心功能）

阶段四：外部访谈
  - 动作：机构去访谈公司的客户、供应商、行业专家
  - 痛点：外部被访谈者表达不准确，可能说出不利于项目的话
  - 特殊点：外部被访谈者与公司关系良好，"可适度培训和要求改进"
  - 现有覆盖：部分（AI Coach有外部访谈场景，但外部培训功能未开发）

阶段五：谈判与协议
  - 动作：term sheet谈判、投资协议签署
  - 现有覆盖：❌ 暂无覆盖
```

## 三、现有系统深度解析

### 3.1 AI路演教练（AI Pitch Coach）— 系统A

**定位**：投融资访谈的AI复盘与准备工具

**核心架构**：

```
音频录音 → ASR转写（词级时间戳）→ LLM深度评判 → 结构化分析报告 → 人机协同审查台 → 锁定输出HTML
```

**已实现的多公司/多机构设计**（V10.x）：

- `institution_id` + `company_id` 双维度隔离
- `institution_registry.py`：机构注册表，短名称模糊匹配
- `partner_profiler.py`：按机构+投资人维度构建画像
- 客户只读Dashboard：外部机构可查看自己的访谈数据
- `briefing_engine.py`：会前简报生成（基于历史记忆）
- 多语言支持：中英文自动检测

**核心数据飞轮**：

```
录音上传 → AI分析 → 审查台人工确认 → 锁定落盘 
→ 错题本（ExecutiveMemory）积累 
→ 下次访谈注入历史记忆（historical_profile）
→ AI分析更精准 → 循环
```

**核心价值验证**：已有真实数据积累（`.executive_memory/`、`.company_profiles/`目录存在），说明系统已在真实访谈场景中运行。

**技术亮点**：

- 词级时间戳锚定（非模糊匹配），实现零误差音频切片
- 两阶段深评引擎（轻量扫描 + 聚焦深评），10靶点耗时50s→10s
- 三道防线审查台（狙击清单→精炼→AI润色）
- ASR三级缓存（内存→磁盘→云端），跨session免费秒开

### 3.2 仓颉FSS（资产台账管理系统）— 系统B

**定位**：投融资部知识资产的双向同步管理 + 智能尽调打包

**已实现且稳定运行的部分**：

**双向同步引擎**：

```
物理文件系统 ←→ Excel台账（投融资部知识资产台账.xlsx）
            ↕
    每个文件夹生成 00_索引.md（文件护照）
```

- 向上扫描：递归发现新文件→写入台账，删除文件→打灰色删除线标"已作废"
- 向下同步：读台账有效记录→在对应目录生成Markdown索引文件
- 有效差异引擎（剥离缝合算法），保留历史变更日志不被覆写

**资产活力雷达**：

- 按更新日期分类：高活（≤30天）、休眠（31-90天）、僵尸（>90天）
- 综合活力分 = max(0, 100 - 僵尸×2 - 休眠×0.5 + 总数×0.1)
- 生成HTML战报，包含趋势折线图、团队贡献排行榜
- 历史趋势沉淀到CSV，追踪团队知识生产力

**已搁置的部分——MatchMaker V4.0**：

- 功能设计：解析机构尽调需求文本 → 匹配文件 → 自动打包发送
- 双引擎设计：AI语义引擎（DeepSeek）+ 本地difflib兜底
- 搁置原因：极度不稳定 + Token消耗过高 + 准确率不达预期

**根本原因分析（重要）**：
MatchMaker失败的核心原因是**用文件名做语义匹配的原材料信息密度不足**。文件名是世界上信息密度最低的字符串，"2024年Q1利润测算表.xlsx"无法可靠地匹配到"近三年财务预测"这个需求。

**文件库结构**（反映真实融资材料体系）：

```
01_标准素材库/
  ├── 00_通用素材库/（公司介绍、历史问答、投资人路演参考）
  ├── 01_财务文件/（固定资产、成本、合同、税务、财报、人员明细）
  ├── 02_法务文件/（股权结构、合规、资质、知识产权）
  ├── 03_业务文件/（产品、市场、项目计划）
  ├── 04_融资专项/（A/B轮协议、股东实缴凭证）
  └── 05_公司内部存档/（会议纪要、股东大会、保密协议）
02_对外发送记录/（每次打包的历史记录，带时间戳）
```

**深层设计意图**（主理人明确表达）：

1. 通过双向同步引擎，让公司每年融资材料有实质迭代进步（而非每次从零开始）
2. 让团队把知识沉淀为可复用的资产
3. 通过活力雷达量化每个人的知识贡献
4. **将知识贡献指标与年终奖挂钩**（这是系统的管理价值核心）

## 四、核心战略判断

### 4.1 这套系统本质上是什么？

> **「为中国中小型创业公司，打造一套融资全生命周期的AI作战操作系统（Fundraising Operating System, FOS）」**

这不是单个工具，是一个**操作系统**：流程 + 数据 + 工具的有机组合。

现状是：几把好锤子，但还没有统一的工具箱。

### 4.2 价值到底大不大？

**结论：非常大，但真正的核心价值还未被完整表述。**

主理人表述的驱动力是：

- 提效打包材料（工具价值）
- 沉淀知识资产（组织管理价值）
- 量化团队贡献（HR管理价值）

但**最核心的价值**是：

> **融资是一场信息博弈。投资机构通过信息获取和验证来降低决策风险；创业公司需要在有限时间窗口内，用高质量、一致性、可信度的信息通过投资机构的信任门槛。**
>
> **这套系统真正在做的，是把创业公司的「信息组织能力」系统化、可沉淀、可迭代。**

三个价值层次：


| 层次       | 价值表述                     | 受益角色    |
| -------- | ------------------------ | ------- |
| **效率价值** | 尽调打包从2小时→5分钟             | 投融资专员   |
| **质量价值** | 融资材料像代码一样持续迭代，每轮融资有实质进步  | 创始人/CFO |
| **组织价值** | 知识贡献量化，让"更新资料"成为可考核的工作指标 | CEO/HR  |

## 五、发现的五个重要盲区

### 盲区一：MatchMaker失败原因是实现方式错了，不是功能本身错了

**失败根因**：文件名作为语义信息的信息密度极低。

**正确解法**：建立文件摘要索引层，分离"入库处理"和"在线匹配"：

```
入库时（一次性，离线，可接受较高成本）：
  文件名 + 文件内容 → LLM生成100字摘要 + 3-5个尽调场景标签
  → 存入台账的"文件摘要"列和"场景标签"列

匹配时（在线，低成本，高稳定性）：
  机构需求文本 → 一次LLM调用，结构化解析为【需求关键词 + 场景类型】
  → 在本地摘要+标签库中做关键词检索/模糊匹配
  → 不再依赖大量LLM调用，稳定且廉价（Token降低90%）
```

### 盲区二：资产组织逻辑与投资机构思维逻辑错位

**公司现有分类逻辑**（行政导向）：

```
01_财务文件 / 02_法务文件 / 03_业务文件 ...
```

**投资机构尽调清单逻辑**（投资逻辑导向）：

```
■ 商业模式验证材料  ■ 财务健康度材料  ■ 团队背景材料  ■ 法律风险材料  ■ 市场空间材料
```

两套分类之间存在语义鸿沟，这正是MatchMaker在用暴力LLM弥合，成本高且不稳定。

**正确解法**：引入双标签体系

- **行政标签**：现有目录结构，用于内部管理
- **尽调场景标签**：商业模式/团队/财务/法律/市场，用于外部响应

文件入库时人工或AI辅助标注场景标签，打包时按场景标签检索，精准且廉价。

### 盲区三：系统缺乏"失败复盘→材料改进"的反向闭环

现在的系统是单向的：**公司 → 整理材料 → 发出去**

实际的融资学习循环是双向的：

- 这轮机构为什么没过？→ 记录原因（毛利率太低/商业模式不清晰/市场空间存疑）
- 这个问题被问了几次？→ 应该在BP里主动解释
- 哪些材料每次都要但总是临时找？→ 应该作为"必备常备材料"标记

**缺失的闭环**：

```
发出材料包 → 机构反馈（通过/否决/追问）→ 记录反馈原因 → 
自动标记哪些材料需更新 → 团队分工改进 → 下轮融资时材料质量提升
```

这恰恰是从第二轮融资开始产生巨大价值的功能。

### 盲区四：外部访谈（阶段四）是被低估的蓝海场景

主理人说"外部被访谈者跟我们关系不错，可以适度培训和要求改进"——这句话非常关键。

**场景**：投资机构去访谈你的大客户A，客户A对你们的评价不好，或者说出了不该说的话 → 直接导致项目挂掉。

**解法**：在外部访谈前，AI系统给客户A提供一个15分钟的"友情提示包"：

- 投资人通常会问哪几类问题
- 建议用什么方式表达合作价值
- 哪些话题应该谨慎回答（避免踩雷）
- 强调哪些合作亮点

这是**合规范围内、对双方都有价值的访谈准备**。市场上几乎没有人系统化解决这个问题。

### 盲区五：最大的潜在用户可能是FA（财务顾问），而非单个创业公司

**FA的痛点**：同时代理8-15家创业公司的融资，要管理8-15套尽调材料、访谈准备、跟进记录——工作量是灾难。

**价值乘数**：

- 单个创业公司：只在融资期间（1-2年）高频使用
- FA机构：常年持续使用，覆盖10+家被投公司
- 支付意愿：FA的收费模式（2%+20%carry）意味着有足够的利润空间付工具钱

**这不是说现在要转型**，而是在产品架构上预留多公司/多项目的设计空间（AI Coach已经在做，FSS还未覆盖）。

## 七、建设优先级路线图

### 第一优先级（现在做，1-2周）

**建设文件摘要索引层**

这是解锁整个系统后续功能的关键基础设施。

```python
# 伪代码：文件入库时触发
def on_file_added_to_ledger(file_path, file_name):
    # 尝试读取文件内容（PDF/Word/Excel等）
    content = extract_text_from_file(file_path)
    # 一次性LLM调用生成摘要（离线，不在关键路径上）
    summary = llm_summarize(content, max_100_chars=True)
    # 推断尽调场景标签（商业模式/团队/财务/法律/市场）
    tags = llm_tag_for_dd_scenario(file_name, summary)
    # 写回台账的"文件摘要"和"场景标签"列
    update_excel_row(file_name, summary=summary, tags=tags)
```

**价值**：完成这一步后，MatchMaker可以基于摘要+标签做本地匹配，Token消耗降低90%，稳定性大幅提升。

### 第二优先级（2-4周）

**基于摘要+标签重做MatchMaker（阶段二尽调响应台）**

```
机构需求文本（自然语言）
    ↓ [一次LLM调用，结构化解析]
结构化需求列表 {场景类型, 关键词, 时间范围}
    ↓ [本地检索，无LLM]
摘要+标签库匹配 → 匹配结果 + 缺失预警
    ↓ [文件复制，无LLM]
发货包 + 发货记录
```

### 第三优先级（并行推进，持续）

- AI Coach阶段三访谈台：继续迭代（已有成熟基础）
- 失败复盘看板MVP：从简单表格开始，记录每次尽调结果和机构反馈原因
- 外部访谈"友情提示包"：轻量版，先用模板+AI生成，验证场景价值

### 暂缓

- 统一UI壳（先让各模块独立跑好，再考虑整合前端）
- 阶段五谈判协议（场景复杂，暂不覆盖）

## 八、更大的战略视野：这可能是一个产品

### 8.1 目标用户金字塔

```
Level 3：FA机构（最高价值）
  - 同时代理10+家公司融资
  - 需要系统管理所有被代理公司的材料+访谈
  - 愿意支付SaaS费用，且是常年用户

Level 2：有融资经验的创业公司
  - 已经历过1轮融资，知道痛在哪
  - 愿意为系统化工具付费

Level 1：首次融资的创业公司（入口用户，最大量）
  - 没有任何融资经验，不知道该准备什么
  - 这个系统可以成为他们的"融资启蒙向导"
```

**核心洞察**：Level 1是最大的市场入口，但变现能力最弱；Level 3是最强的变现用户，但获取成本高。产品路径应该是从Level 1教育市场，从Level 3获取营收。

### 8.2 与AI Coach的关系：相互增强的飞轮

```
AI Coach积累的数据：
  - 各机构关注哪类风险点（partner_profiler）
  - 高管在哪类问题上容易翻车（executive_memory错题本）
  - 哪些行业/赛道被问的问题模式（行业基准）

→ 可以告诉FSS：
  - 该机构通常会要这类材料（提前预备尽调包）
  - 这类问题对应的支撑材料是什么（自动关联）
  - 材料中的哪个数据点容易被质疑（提前标注和说明）

FSS积累的数据：
  - 哪些材料被要求最频繁（高频材料 = 高优先级维护）
  - 哪些材料总是缺失（指导公司补齐材料库）
  - 每次发货后机构的反馈结果（复盘输入）

→ 可以告诉AI Coach：
  - 历史问答库（QA材料）帮助访谈准备更精准
  - 材料健康度影响访谈中的答题自信度
```

**结论**：两个系统之间存在真实的数据协同价值，整合后的系统远大于各部分之和。

## 九、技术架构现状快照（供AI会诊参考）

### AI Coach技术栈

```
前端：Streamlit
转写：阿里云DashScope（主）+ 硅基流动（备）
LLM：DeepSeek（主）+ Kimi + Qwen
存储：本地文件系统（JSON + HTML + MP3）
缓存：三级缓存（内存 → 磁盘MD5 → 云端）
测试：pytest（513 passed，全量Mock外部API）
多公司：institution_id + company_id 双维度隔离
数据沉淀：analytics_exporter.py，{stem}_analytics.json
```

### 仓颉FSS技术栈

```
前端：tkinter桌面GUI（打包为EXE交付，零环境配置）
Excel处理：openpyxl（严禁pandas，保证轻量化）
LLM：DeepSeek（MatchMaker，已搁置）
文件匹配：difflib.SequenceMatcher（本地降级引擎）
测试：unittest，混沌工程级（自动构建/销毁测试沙盒）
单公司设计：当前仅支持单公司，无多公司隔离
```

### 关键技术债务

1. FSS无摘要索引层 → MatchMaker不可用
2. FSS无多公司设计 → 无法扩展为产品
3. 两套系统无数据互通 → 价值孤岛
4. AI Coach无与FSS的材料库关联 → 访谈准备缺少材料支撑
