## 1. 先说结论：这套系统现在到底是什么

FOS（Fundraising Operating System，融资作战操作系统）不是一个单点工具，而是一套围绕**融资全生命周期**组织起来的 AI 工作系统。它的核心目标不是“做一个会聊天的大模型壳子”，而是把一家创业公司在融资过程中最关键、最高频、最容易失控的几个环节，变成**可记录、可分析、可复用、可协同、可回放**的工程系统。

目前 FOS 由两个主要系统组成：

1. **System A：AI_Pitch_Coach**
   - 面向路演、机构访谈、高管访谈、复盘、会前准备、机构画像、融资作战室。
   - 主技术栈：**Streamlit + Python + 本地文件系统 + 云端 ASR/LLM API**。
   - 当前是 FOS 的主战场，也是当前最成熟、最有业务价值的部分。

2. **System B：AI_CangJie_FSS**
   - 面向尽调材料与知识资产的整理、台账维护、索引同步、资产健康分析。
   - 主技术栈：**tkinter + Python + Excel + 本地文件系统**。
   - 当前承担“资产底座”和“材料秩序化”职责。

两者通过一个极轻量的数据桥对接：

```text
D:\AI_Workspaces\.fos_data\
└── asset_index.json
```

这意味着：**FOS 当前不是一个云原生 SaaS 集群，而是一个本地优先、文件系统优先、工程稳态优先的复合系统**。它的价值不是来自“技术名词新不新”，而是来自它对真实融资流程的贴合程度。

## 2. 为什么这个系统有价值

### 2.1 它解决的不是抽象 AI 问题，而是实际融资作业问题

对中小型科技公司来说，融资过程里最贵的不是模型调用费，而是：

- 创始人/高管在访谈里说错一句话，导致机构信心下降
- 同一类问题反复被问住，没有形成组织记忆
- 材料散落在文件夹里，找材料靠人脑和微信
- 见机构之前没有系统性准备，只靠临场反应
- 复盘结果不沉淀，下次还是从零开始

FOS 当前已经把其中最关键的一条链路做得比较完整：

**录音 -> 转写 -> 风险识别 -> 结构化复盘 -> 人工审查 -> 报告固化 -> 数据沉淀 -> 记忆积累**

这条链路的价值在于，它把“经验”从个人头脑中抽出，变成了**组织级可复用资产**。

### 2.2 它不是泛化办公助手，而是强场景作战系统

这套系统的设计哲学不是“万能 Copilot”，而是“高价值垂直场景深挖”：

- 对象不是泛用户，而是**融资场景中的创始人、高管、FA、材料管理员**
- 输入不是随机聊天，而是**音频、QA 资料、公司背景、机构历史、资产台账**
- 输出不是泛泛建议，而是**词级锚定的风险点、扣分理由、标准口径、可导出的报告**

因此，系统价值主要体现在：

- **减少翻车**：降低关键访谈中的表达失误
- **提升准备度**：会前知道机构最可能问什么
- **形成数据飞轮**：同一个公司、同一个高管、同一家机构，越用越准
- **增强可协同能力**：让材料、复盘、画像、进度不再散落

## 3. 系统设计哲学

如果让另一个 AI 来评估下一步方向，必须先知道这套系统的设计哲学，否则容易给出错误的“炫技式建议”。

### 3.1 设计哲学一：先稳态，再智能

这套系统过去不是从“Agent 很酷”出发，而是从“**一条关键任务链必须跑通且可复盘**”出发。

所以你会看到很多工程选择都偏保守：

- 用 **Pydantic 契约**，而不是无结构自由输出
- 用 **本地 JSON/HTML 持久化**，而不是先上复杂数据库
- 用 **关键词命中 + 规则匹配**，而不是一上来就向量库
- 用 **审查台锁定导出**，而不是让 AI 直接产出终稿
- 用 **原子写入、静默降级、测试优先**，而不是“先能跑再说”

这不是“技术不先进”，而是因为系统面对的是**业务后果非常重**的场景。

### 3.2 设计哲学二：Human-in-the-Loop 不是补丁，而是主架构

当前系统从 V3 开始就明确不是“AI 自动化替代人”，而是：

> AI 出结构化初稿 -> 人审查 -> 人锁定 -> 系统导出最终结果

这意味着：

- 人工不是系统 bug 的补救者，而是流程设计中的一等公民
- 所有关键输出必须支持人工审查和重写
- 系统应优先追求**可编辑性**、**可解释性**、**可回放性**

### 3.3 设计哲学三：文件系统优先于重型基础设施

截至 2026-04-16，FOS 仍然是一个**本地优先**系统：

- 错题本：JSON 文件
- 草稿：JSON 文件
- analytics：JSON 文件
- 报告：HTML 文件
- 资产桥：`asset_index.json`
- Pipeline：JSON

这带来的好处是：

- 部署轻
- 可移植
- 易审计
- 易打包
- 易回退

也意味着：

- 当前还没有真正意义上的中心化多租户后端
- 一些“租户隔离”“跨端同步”“并发冲突”仍依赖工程约束而非数据库事务

### 3.4 设计哲学四：只有与业务真实路径对齐的记忆，才值得保留

这套系统里的“记忆”不是为了模拟人脑，而是为了形成**可用的业务偏差修正机制**。

也就是说：

- 记忆不是聊天记录全文堆积
- 记忆不是向量库里一堆 embedding
- 记忆是“某个高管在某类问题上容易犯的错误”和“建议口径”的沉淀

因此当前记忆结构非常克制：`company_id + tag(interviewee)` 分桶，Top N 注入 Prompt。

## 8. 当前工程架构全景（截至 2026-04-16）

下面从层次结构梳理整个系统，方便另一个 AI 做下一步架构判断。

### 8.1 展示层：`app.py`

当前前端仍是 **Streamlit 单体控制台**，主要承担：

- 项目档案选择
- API 配置与环境自检
- 录音上传
- 批量录音处理
- QA 文件上传
- 审查台
- 数据中台/看板
- 会前简报
- 融资作战室
- 投资人匹配
- 融资 Pipeline
- **2026-04-17 起**：批量页「提取文字稿」区支持可选 ASR 轻量润色、被访谈人与「发言人 N」映射（展示 + `explicit_context.session_notes` 提示）；审查台锁定后 Agentic 双栏（动态画布 + 教练区动作按钮与会前简报一键触发）；数据中台 Tab「机构画像」内 GitHub 同步告警按 **analytics / institutions** 分通道展示，并含 **「拉取本公司 analytics」** 显式下载（见 **§17.5** 与 `2026-04-17_coach_data_上传下载清单与操作流程.md`）

优点：

- 交互效率高
- 改功能成本低
- 非前端团队也能快速迭代

缺点：

- `app.py` 仍然是一个巨型文件
- UI 状态、业务入口、路由逻辑仍然耦合较重
- 从长期看不是最理想的可维护结构

### 8.2 编排层：`job_pipeline.py` + LangGraph

这是当前的关键演进层。

#### 旧模式

`job_pipeline.run_pitch_file_job(...)` 负责：

- ASR
- 脱敏
- 调 `evaluate_pitch`
- 写 JSON
- 调 `report_builder`

#### 新模式

当开关打开时：

- `job_pipeline` 只负责外围任务流水线
- 核心评估步骤转入 `agent_runner -> agent_workflow -> agent_nodes`

这意味着：当前系统正处于一种**双栈共存、渐进迁移**状态。

### 8.3 核心推理层：`llm_judge.py`

这是系统最重要的业务智能内核。

当前主要职责：

- 构建 Prompt
- 语言检测
- 阶段一扫描
- 阶段二深评
- JSON 抢救
- 结果组装
- 错题本提炼（`distill_executive_memory_from_diff`）

当前问题：

- 虽然已拆阶段函数，但 Prompt 体系仍然非常重
- 很多核心业务知识仍然编码在超长系统提示中
- 这部分仍是整个系统最需要持续治理的技术债区域

### 8.4 记忆层：`memory_engine.py`

当前记忆体系仍然是**轻量 JSON 持久化**：

- 根目录：`.executive_memory/`
- 一级桶：`company_id`
- 二级桶：`tag`（通常是被访谈高管）
- 内容：`ExecutiveMemory`

特点：

- 没有向量库
- 没有复杂召回
- 有防噪门
- 有命中计数和更新时间
- 有注入 Prompt 的 Top N 逻辑

这是一个**非常朴素但有效的 Episodic Memory 实现**。

### 8.5 资产层：`asset_bridge.py` + `AI_CangJie_FSS`

当前资产协同模式：

- FSS 负责维护资产台账和生成 `asset_index.json`
- AI Coach 通过 `asset_bridge.load_asset_index()` 读取全局资产索引
- 当前匹配方式仍然是**关键词命中**

这是一个有意识保持轻量的设计：

- 不搞向量数据库
- 不搞在线 API 桥
- 不搞复杂同步协议

当前阶段，这种克制是合理的。

### 8.6 报告层：`report_builder.py`

系统的差异化价值之一就在这里：

- 根据词级索引回写 `original_text`
- 根据时间戳生成音频片段
- 把“结论、原文、音频”绑定到一起

这使 FOS 不是一个“抽象评论工具”，而是一个**证据链完整的复盘系统**。

### 8.7 数据沉淀层：analytics / registry / dashboard

这部分把一次次分析结果转化为更高层的可用资产：

- 机构画像
- 个人成长
- 行业基准
- 会前简报
- 投资人匹配
- Pipeline

这意味着系统已经不是“单次任务工具”，而是在逐渐形成一个**融资行为数据底座**。

## 9. 当前系统的真实优势

如果要让另一个 AI 判断系统价值，不能只看代码结构，必须看“为什么这套东西难得”。

### 9.1 优势一：业务贴脸

这套系统对融资高管访谈的理解不是泛化的，而是非常贴近真实工作流：

- 路演
- 尽调
- 高管访谈
- 外部访谈
- 会前准备
- 机构历史

这不是一个“找一个行业壳套在通用 Agent 上”的产品，而是从真实作业链里长出来的。

### 9.2 优势二：证据链完整

很多 AI 分析系统最大的弱点是：说得像回事，但无法追溯。

FOS 当前的结构使它可以把：

- 原音频
- 转写
- 词级索引
- 风险点
- 扣分
- 原文
- 音频切片
- 审查修改

串成一条证据链。

### 9.3 优势三：有组织记忆，而不是一次性调用

系统不是“每次重新问模型”，而是在逐步形成：

- 某个高管的历史坑
- 某类机构爱问什么
- 某家公司长期薄弱点
- 某些资产在何场景高频被需要

这才是 FOS 真正可能形成护城河的地方。

### 9.4 优势四：重构方式克制

本轮 LangGraph 改造的价值不在于“用了 LangGraph”，而在于：

- 没有掀翻旧系统
- 没有一次性大重写
- 没有牺牲测试稳定性
- 没有引入超出当前组织能力的基础设施

这是非常重要的工程判断。

## 10. 当前系统的真实问题与约束

另一个 AI 若要讨论下一步方向，必须正视这些问题。

### 10.1 问题一：`app.py` 仍然过大

当前 UI 层承担了太多职责：

- 页面
- 状态
- 入口
- 审查
- 一部分业务编排

这会在后续功能继续扩张时成为维护瓶颈。

### 10.2 问题二：Prompt 依然偏重

即便 Week 2 已经把评估拆成阶段函数，
**大 Prompt 拼接** 仍然主要存在于 `llm_judge.py`。

这意味着：

- 可维护性仍然受限
- 规则治理成本高
- 行为回归难度大

### 10.3 问题三：租户隔离当前是“工程协议”，不是“基础设施隔离”

虽然 Week 3 已经加了 `agent_tenant.py` 和 `memory_io_enabled`，
但当前隔离本质上仍然依赖：

- `tenant_id` 传对
- `memory_company_id` 约束生效
- 代码里遵守闸门

它还不是数据库级、服务级、ACL 级的强隔离。

### 10.4 问题四：资产索引仍是全局只读

这是有意为之，但要清楚其含义：

- 当前资产索引不是严格按租户分区
- 适合内部使用
- 若未来真的做 B2B SaaS，多租户资产隔离必须升级

### 10.5 问题五：记忆闭环仍然是“半闭环”

当前已有：

- 审查台锁定后收割错题本
- 图尾 telemetry
- 图开头检索记忆

但还没有：

- 真正统一的图内“写记忆”流程
- 异步队列/审核机制
- 多来源记忆融合策略

这说明当前系统仍处于**从“工具逻辑”向“Agentic 系统”演进的中段**。

#### Week 4（最小脱敏链路）：进入 LLM 前的输入清洗

已完成：

- 新增：
  - `src/agent_sanitize.py`
  - `tests/test_agent_sanitize.py`
  - `tests/test_week4_sanitize_node.py`
- `AgentState` 扩展出：
  - `sanitized_qa_text`
  - `sanitization_meta`
- LangGraph 新增节点：
  - `sanitize_inputs`
- 当前图更新为：

```text
ingest
  -> retrieve_memory
  -> sanitize_inputs
  -> prepare_eval
  -> phase1_scan
  -> phase2_report
  -> finalize
  -> feedback_telemetry
  -> END
```

当前策略：

- 仅对**进入 LLM 的 QA 文本**做最小脱敏
- **原始 `words` 不改**
- **原始 `qa_text` 不改**
- 默认走稳定的**正则脱敏后端**
- 已安装 `presidio-analyzer` / `presidio-anonymizer`，但当前仅在显式开启 `USE_PRESIDIO_SANITIZER=1` 时尝试使用，以避免本地环境初始化过慢导致主链路阻塞

意义：

- 系统开始具备正式的“进入模型前合规清洗”挂点
- 为后续 Week 4 的更强脱敏、Week 5 的上下文增强提供了安全边界

## 12. 当前最值得讨论的下一步方向

这是交给另一个 AI 讨论时最应该聚焦的部分。

### 12.1 方向一：是否进入 Week 4（脱敏与合规链路）

推荐优先级：**高**

原因：

- 当前已有租户与记忆雏形
- 若继续往真实 SaaS 演进，脱敏与审计会迅速变成硬约束
- `presidio-analyzer` 是自然下一步

讨论问题：

- 脱敏发生在什么层？
- 是只对进 LLM 的文本脱敏，还是全链路字段级脱敏？
- 如何兼容词级锚点与脱敏替换？

### 12.2 方向二：是否让 `asset_hits` 真正进入 Prompt

推荐优先级：**高**

原因：

- Week 3 已经把资产命中挂进 state
- 现在它只是可观测信息，还没有真正影响推理
- 若能谨慎注入，会提升“建议不空泛”的程度

讨论问题：

- 注入在哪里最安全：`prepare_eval` 前还是 phase1/phase2 之间？
- 注入什么形式：摘要片段还是文件列表？
- 如何避免 prompt 膨胀？

### 12.3 方向三：是否做统一的记忆写入策略

推荐优先级：**中高**

原因：

- 当前读已经图化，但写还分散
- 长期看，读写不统一会成为数据一致性问题

讨论问题：

- 图尾是否应该只生成 `pending_memory_events`，再由单独 worker 入库？
- 如何与审查台锁定路径去重？
- 什么样的 risk point 才配进入 Episodic Memory？

### 12.4 方向四：是否继续拆 `app.py`

推荐优先级：**中**

原因：

- 技术债明确存在
- 但短期产品扩展与 Agentic 方向相比，不一定是最优先

讨论问题：

- 是拆页面组件，还是先拆服务层？
- 在 Streamlit 体系下，最小成本的治理方式是什么？

### 12.5 方向五：是否把真正的 B2B SaaS 多租户后端纳入中长期规划

推荐优先级：**战略级，不是立即实施**

原因：

- 当前系统更像“本地优先的高价值内部作战系统”
- 若要面向外部商业化，最终仍要走中心化租户后端

讨论问题：

- 什么时候值得从 JSON + 文件系统迁到数据库？
- 哪些能力必须先成熟，才值得上后端重构？

### 17.1 Agentic UI（Streamlit，无新 UI 库）

- **范围说明**：当前映射 UI 以**第一条录音 / batch_iv_0** 为主路径；多文件逐条扩展可后续做。

### 17.3 数据飞轮 P0：`coach_data` 上 analytics 必达（工程修复，非新业务能力）

**现象（修复前）**：远端常只见 `institutions/institutions.json`，`analytics/` 不出现或误以为「没上传」。

**根因（已修）**：

1. **`app.py` `_v3_finalize_stem`**：`github_sync_analytics` 曾被包在 `if _iid:` 内，**无机构 ID 时整条不上传 analytics**。现改为：**只要 `analytics_path` 存在就尝试 `github_sync_analytics`**；`github_sync_institutions` 仍在有 `_iid` 时调用。
2. **`src/github_sync.py` `sync_analytics`**：GitHub Contents API 路径须 URL-safe；**中文 `company_id` 与中文文件名**会导致 `UnicodeEncodeError`，上传静默失败。现改为上传路径使用 **ASCII 安全目录名 + ASCII 安全文件名（含原名校验用 hash 后缀）**；**本地文件路径与内容不变**。
3. **状态文件误导**：`push_file` 成功时原先用单一 `last_success` 覆盖，**institutions 成功会掩盖 analytics 失败**。现 `github_sync_status.json` 增加 **`channels.analytics` / `channels.institutions`** 分通道计数与时间戳；`get_sync_status()` 返回 `analytics` / `institutions` 子对象（含 `needs_alert`）。旧版顶层字段仍写入以保持兼容。
4. **`app.py` `_render_sync_status_alert`**：优先展示 **analytics** 通道告警；institutions 单独 warning。

**核验**：锁定或手动 `sync_analytics` 后，查 `coach_data` 仓库 commit 应出现 `sync analytics: <本地原名>`，且 tree 中存在 `analytics/<safe_company>/...json`。
